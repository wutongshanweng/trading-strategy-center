# 交易系统统一数据库设计

## 一、设计原则

### 1. 具体合约，不搞合成

| 资产 | 示例 | 说明 |
|:----|:----|:----|
| 期货 | RB2509, RB2510 | 具体可交易合约 |
| 期权 | IO2509-C-3700 | 标的+月份+类型+行权价 |
| 股票 | 600019 | 股票本身就是具体的 |
| 宏观 | GDP_2025Q3, CPI_202606 | 具体发布期 |

### 2. 品种/合约两层结构

```
products（品种层）:  RB, FG, CU, IO     ← 策略归类、跨市场映射
    ↓ 一对多
symbols（合约层）:   RB2509, RB2510     ← 实际价格数据
                    IO2509-C-3700
```

### 3. 多周期设计

| 资产 | 周期 | 存储策略 |
|:----|:----|:--------|
| **期货** | M5, M15, M30, H1, H4, D1, W1, M1 | 统一 kline 表，timeframe 字段区分 |
| **股票** | D1（日线） | 统一 kline 表 |
| **期权** | D1（日线），Greeks/IV 在 options_daily | 日线 + Greeks 分开 |
| **宏观** | 按发布频率（月/季） | macro_data 表 |

D1/W1/M1 可以从 D1 聚合，但存起来查询更快，不需要每次算。

### 2.3 K线表 kline（统一多周期）

所有资产、所有周期都在一张表里。

```sql
CREATE TABLE kline (
    datetime        TIMESTAMP NOT NULL,          -- bar 开始时间
    symbol_id       INTEGER NOT NULL REFERENCES symbols(symbol_id),
    timeframe       VARCHAR NOT NULL,            -- 'M5','M15','M30','H1','H4','D1','W1','M1'
    open            DECIMAL(12,4),
    high            DECIMAL(12,4),
    low             DECIMAL(12,4),
    close           DECIMAL(12,4),
    volume          BIGINT,
    amount          BIGINT,                      -- 成交额
    open_interest   BIGINT,                      -- 持仓量（期货/期权）
    -- 期货结算价
    settlement      DECIMAL(12,4),
    pre_settlement  DECIMAL(12,4),
    created_at      TIMESTAMP DEFAULT now(),
    PRIMARY KEY (datetime, symbol_id, timeframe)
);

CREATE INDEX idx_kline_sym ON kline(symbol_id, timeframe, datetime);
CREATE INDEX idx_kline_date ON kline(datetime);
```

**说明：**
- 期货存 M5/M15/M30/H1/H4/D1/W1/M1 八个周期
- 股票只存 D1（timeframe = 'D1'）
- 期权只存 D1
- M5 是最小粒度，H1/H4/D1 可以从 M5 聚合
- W1/M1 从 D1 聚合（不需要实时精度）
- 聚合结果存起来，避免每次查询都重算

**数据样例：**

| datetime | symbol_id | timeframe | open | close | volume |
|:---------|:---------|:---------|:----|:-----|:------|
| 2026-06-17 09:00:00 | RB2509 | D1 | 3160 | 3170 | 185000 |
| 2026-06-17 09:05:00 | RB2509 | M5 | 3160 | 3165 | 12500 |
| 2026-06-17 09:10:00 | RB2509 | M5 | 3165 | 3162 | 8900 |
| 2026-06-17 09:00:00 | RB2509 | H1 | 3160 | 3175 | 52300 |
| 2026-06-15 00:00:00 | RB2509 | W1 | 3120 | 3170 | 850000 |
| 2026-06-01 00:00:00 | RB2509 | M1 | 3080 | 3170 | 3200000 |

**聚合策略：**

```
源数据 → 拉M5（最小粒度）
  ├→ 聚合 → M15/M30/H1/H4
  └→ D1    → 拉M5或从行情接口直接拿
W1    → 从D1聚合（周一开盘到周五收盘）
M1    → 从D1聚合（月首日开盘到月末收盘）
```

## 三、数据库选型：DuckDB

```python
# 安装
pip install duckdb

# 使用
import duckdb
conn = duckdb.connect('data_center.db')

# 建表、插入、查询全部SQL
conn.execute("CREATE TABLE ...")
conn.execute("INSERT INTO kline VALUES (...)")
result = conn.execute("SELECT * FROM kline WHERE ...").fetchdf()
```

**为什么用 DuckDB？**

| 场景 | SQLite | DuckDB |
|:----|:------|:------|
| 跨品种相关性计算 | 慢（行存储） | 快（列存+向量化） |
| 多周期聚合（M5→H1→D1） | 需循环 | SQL一行搞定 |
| 千万级时间序列查询 | 慢 | 毫秒级 |
| 单文件部署 | 是 | 是 |
| 读 Parquet/CSV | 无 | 原生 |
| Python 集成 | 标准 | `fetchdf()` 直接转 pandas |

**数据量预估：**
- 期货：3品种 × 8周期 × 250天 × 5年 = ~630万行 → DuckDB轻松
- 股票：2000只 × D1 × 250天 × 5年 = 500万行
- 期权：活跃合约有限
- 总量在千万级别，DuckDB 几十MB到几百MB

## 六、说明

### 1. W1/M1 周期

每周线和每月线不从原始接口拉，而是从 D1 聚合：
- W1：周一开盘价 → 周五收盘价，聚合该周日线
- M1：月首开盘价 → 月末收盘价，聚合该月日线
- 聚合结果存入 kline 表，timeframe = 'W1'/'M1'

### 2. 主力合约管理

- `is_main` 字段标记当前主力
- 每天开盘前检查成交量/持仓量，自动更新
- 换月记录写入 `futures_main_switches`

### 3. 期权特殊性

- 没有"主力合约"概念，只有不同行权价
- 数据处理以日线为主（D1）
- Greeks/IV 是核心，比价格更重要
- 到期的合约直接标 status = 'expired'

### 4. 数据去重

DuckDB 的 `PRIMARY KEY (datetime, symbol_id, timeframe)` 天然去重，重复拉取不会产生重复数据。
